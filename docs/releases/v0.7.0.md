# GamuCare AI 0.7.0 — Ampliacion y evaluacion del RAG

## Objetivo

La version 0.7.0 convierte el asistente documental en un componente medible. La mejora no se limita a incorporar mas documentos: modifica la ingesta, la recuperacion, la trazabilidad de fuentes, el tratamiento de conversaciones y la forma de comprobar la calidad del sistema.

El modelo generativo sigue sin decidir permisos ni acceso a informacion. Los filtros de propietario, mascota y visibilidad se aplican antes de consultar Qdrant.

## Cambios funcionales

### Base documental ampliada

Se incorporan documentos resumidos y versionados sobre:

- Vacunacion de perros y gatos.
- Viajes desde Espana a la Union Europea.
- Entrada en la UE desde terceros paises.
- Control individualizado de parasitos.
- Enfermedades transmitidas por vectores.
- Peso, condicion corporal y condicion muscular.
- Cuidados preventivos de pacientes senior.
- Seguimiento de enfermedad renal cronica.

Las fuentes principales son WSAVA, ESCCAP, MAPA, Union Europea, AAHA e IRIS. Los textos incluidos son resumenes propios, no copias integrales de los documentos originales.

### Respuestas con identificadores de fuente

Los fragmentos recuperados reciben identificadores `F0`, `F1`, `F2`, etc. El prompt exige que las afirmaciones relevantes se relacionen con esos identificadores.

- `F0` se reserva para la ficha estructurada de una mascota consultada por su propietario o por la clinica.
- `F1` en adelante representan documentos o eventos recuperados desde Qdrant.
- La interfaz muestra el mismo identificador junto a cada fuente.

### Contexto conversacional

El asistente utiliza hasta seis mensajes anteriores de la misma sesion. Esto permite preguntas de seguimiento sin mezclar conversaciones pertenecientes a otra mascota.

### Deteccion preventiva de posibles urgencias

Se incorpora una clasificacion determinista por terminos de alarma. Cuando se detecta una expresion como dificultad respiratoria, convulsiones, intoxicacion o imposibilidad para orinar, el prompt obliga a priorizar la recomendacion de contactar con una clinica veterinaria.

Este mecanismo no diagnostica ni intenta determinar la gravedad real del caso.

### Diagnostico de recuperacion visible

Cada respuesta incluye:

- Confianza de recuperacion: baja, media o alta.
- Motivo de la clasificacion.
- Numero de candidatos revisados.
- Numero de fragmentos seleccionados.
- Puntuacion del mejor resultado.
- Categorias, especie, ambito y filtros aplicados.
- Indicador de posible urgencia.

La confianza corresponde a la calidad de la recuperacion, no a la certeza clinica de la respuesta.

### Panel Calidad RAG

Los perfiles `clinic` y `staff` disponen de una nueva pantalla con:

- Disponibilidad de Qdrant.
- Numero de vectores.
- Documentos ingeridos.
- Cobertura por categoria.
- Distribucion por nivel de confianza de la fuente.
- Ultima ingesta.
- Resultado de la ultima evaluacion.
- Casos fallidos que requieren revision.

Solo el perfil `clinic` puede lanzar una nueva evaluacion desde la interfaz.

## Cambios tecnicos

### Analisis determinista de consulta

Nuevo modulo:

```text
backend/app/services/rag_intelligence.py
```

Identifica de forma explicable:

- Categoria probable.
- Especie mencionada.
- Ambito geografico.
- Palabras relevantes.
- Terminos de posible urgencia.

### Recuperacion y reranking

El proceso pasa a ser:

```text
Pregunta
  -> analisis determinista
  -> embedding con Ollama
  -> recuperacion amplia en Qdrant
  -> eliminacion de duplicados
  -> reranking semantico + lexico + metadatos
  -> umbral minimo
  -> contexto con identificadores de fuente
  -> generacion con Ollama
```

La puntuacion final combina:

- 78 % similitud densa.
- 12 % coincidencia lexica.
- Bonificaciones limitadas por categoria, especie, ambito y fuente oficial.

La formula es deliberadamente sencilla para poder explicarla y probarla.

### Ingesta mejorada

Los documentos se fragmentan respetando encabezados Markdown y aplicando un solapamiento pequeno. Se añaden indices de payload en Qdrant para los campos de filtrado mas utilizados.

Los metadatos incorporados son:

- Idioma.
- Audiencia.
- Tipo de fuente.
- Nivel de confianza.
- Etiquetas.
- Fecha de revision.
- Categoria y ambito geografico.

### Evaluacion versionada

Nuevo dataset:

```text
data/evaluation/rag_cases_v1.json
```

Incluye 20 consultas:

- Preguntas documentales con respuesta.
- Preguntas fuera de alcance.
- Consultas sobre historiales ficticios recurrentes.

Metricas calculadas:

- Retrieval hit rate.
- Mean Reciprocal Rank (MRR).
- Acierto de categoria.
- Acierto de fuente.
- Acierto de paciente y tipo de evento.
- Exactitud al rechazar preguntas sin contexto.
- Latencia media, P50 y P95.
- Presencia de citas y decision de groundedness cuando se activa la generacion.

La evaluacion automatica no sustituye una revision humana de fidelidad, claridad y utilidad clinica.

## API nueva

```text
GET  /api/v1/rag/status
GET  /api/v1/rag/evaluation/latest
POST /api/v1/rag/evaluation/run
```

El `POST` acepta:

```json
{
  "with_generation": false
}
```

El modo de recuperacion es mas rapido. El modo con generacion tarda mas porque consulta al LLM para todos los casos.

## Base de datos

Migracion incluida:

```text
0006_rag_quality
```

Cambios:

- Ampliacion de `rag_documents` con gobierno de metadatos.
- Nueva tabla `rag_evaluation_runs`.
- Persistencia de metricas y detalle de cada ejecucion.

## Variables de entorno

Nuevas o modificadas:

```text
RAG_MIN_SCORE=0.42
RAG_CANDIDATE_K=24
RAG_TOP_K=6
RAG_EVALUATION_DATASET=/app/data/evaluation/rag_cases_v1.json
```

El cambio de umbral se aplica despues del reranking, no directamente sobre la similitud densa de Qdrant.

## Actualizacion desde 0.6.0

1. Copiar la version sobre la carpeta existente.
2. Conservar `.env`.
3. No eliminar volumenes.
4. Ejecutar:

```powershell
Set-ExecutionPolicy -Scope Process Bypass
.\scripts\upgrade-v0.7.ps1
```

El script:

1. Reconstruye API y frontend.
2. Aplica Alembic hasta `0006_rag_quality`.
3. Reindexa documentos e historiales.
4. Ejecuta la evaluacion de recuperacion.
5. Ejecuta las pruebas del backend.

## Impacto sobre datos existentes

- No se eliminan propietarios, mascotas, historiales, planes, pagos ni alertas.
- Qdrant se reconstruye para aplicar el nuevo fragmentado y los nuevos metadatos.
- Los registros previos de `rag_documents` se conservan y actualizan cuando coincide el hash del fichero.
- Se crea una tabla nueva para almacenar evaluaciones.

## Pruebas

Validacion realizada antes de empaquetar:

- 31 pruebas de backend superadas.
- Compilacion Python completada.
- Compilacion React, TypeScript y Vite completada.
- Comprobacion de ausencia de URLs NPM internas.
- Migracion Alembic incluida y encadenada a `0005_preventive_alerts`.

No se ha ejecutado Docker Compose completo en el entorno de generacion. La evaluacion real con los embeddings de Ollama debe ejecutarse en Docker Desktop.

## Limitaciones conocidas

- El reranking no utiliza un modelo cross-encoder por las restricciones de la GPU de 6 GB.
- La deteccion de urgencias se basa en terminos y puede producir falsos positivos o no detectar expresiones no contempladas.
- Las metricas automaticas verifican recuperacion y citas, pero no demuestran correccion clinica.
- La evaluacion con generacion usa el mismo modelo local y debe complementarse con revision humana.
- Los documentos son resumenes controlados; no sustituyen la consulta de la fuente oficial vigente.

## Reversion

Para volver al codigo 0.6.0 conservando la base de datos:

1. Realizar una copia de seguridad de PostgreSQL.
2. Ejecutar con el codigo 0.7.0:

```powershell
docker compose -f docker-compose.yml -f docker-compose.gpu.yml exec api alembic downgrade 0005_preventive_alerts
```

3. Restaurar los ficheros de la version 0.6.0.
4. Reconstruir API y frontend.
5. Reindexar Qdrant con el script de la version restaurada.
