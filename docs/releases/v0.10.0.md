# GamuCare AI 0.10.0 - Seguridad y auditoria

Fecha: 2026-07-16

## Objetivo

Esta version refuerza la autenticacion, el control de sesiones, la proteccion de datos y la trazabilidad de las operaciones. No cambia el comportamiento clinico del RAG ni el contenido de los planes, pero anade controles transversales a toda la aplicacion.

## Cambios funcionales

### Acceso y contrasenas

- Las contrasenas nuevas deben tener al menos 12 caracteres, mayuscula, minuscula, numero y caracter especial.
- Las cuentas creadas desde **Clientes** o mediante la importacion Wakyma reciben una contrasena temporal y deben cambiarla antes de usar el resto de la aplicacion.
- Tras cinco intentos fallidos, la cuenta queda bloqueada durante 15 minutos. Ambos valores son configurables.
- Los usuarios pueden cambiar su contrasena desde **Seguridad**.

### Sesiones

- El token de acceso dura 30 minutos por defecto.
- La sesion se renueva con una cookie HttpOnly que no es accesible desde JavaScript.
- Cada renovacion rota el token anterior y guarda en PostgreSQL solo su hash.
- El usuario puede consultar y revocar sus sesiones.
- Cambiar la contrasena invalida las sesiones anteriores.

### Auditoria

Se registran, entre otras, estas operaciones:

- Inicio de sesion correcto, fallido o bloqueado.
- Cambio de contrasena, renovacion y cierre de sesion.
- Alta, modificacion, desactivacion y reactivacion de clientes y mascotas.
- Registro de eventos clinicos.
- Asignacion, cambio, cancelacion y renovacion de planes.
- Cambios de cuotas, pagos y prestaciones.
- Transiciones y reconstruccion de avisos preventivos.
- Validaciones e importaciones Wakyma.
- Accesos denegados y peticiones de escritura fallidas.

La pantalla **Auditoria** permite filtrar por usuario, accion, entidad, resultado y fechas. `clinic` y `staff` pueden consultar el registro; solo `clinic` puede exportarlo a CSV.

## Minimizacion de datos

La auditoria no almacena contrasenas, tokens ni cookies. Los correos se enmascaran, los datos de contacto se sustituyen y el texto clinico libre no se duplica en la tabla de auditoria. Los logs tampoco registran cuerpos HTTP ni cabeceras de autorizacion.

## Cambios tecnicos

### Base de datos

Migracion Alembic: `0008_security_audit`.

Cambios en `users`:

- `must_change_password`
- `failed_login_attempts`
- `locked_until`
- `password_changed_at`
- `token_version`

Tablas nuevas:

- `refresh_sessions`
- `audit_logs`

### API

Endpoints nuevos o ampliados:

- `POST /api/v1/auth/refresh`
- `POST /api/v1/auth/change-password`
- `GET /api/v1/auth/sessions`
- `DELETE /api/v1/auth/sessions/{session_id}`
- `GET /api/v1/audit`
- `GET /api/v1/audit/stats`
- `GET /api/v1/audit/export.csv`

### Frontend

Pantallas nuevas:

- `/change-password`
- `/security`
- `/audit`

El token de acceso se guarda en `sessionStorage`; el token de renovacion permanece en cookie HttpOnly. El cliente renueva una vez la sesion cuando recibe un `401` y repite la peticion original.

### Nginx

Se anaden CSP, `X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy` y propagacion de `X-Request-ID`.

## Variables nuevas

```text
ACCESS_TOKEN_EXPIRE_MINUTES=30
REFRESH_TOKEN_EXPIRE_DAYS=7
REFRESH_COOKIE_NAME=gamucare_refresh
REFRESH_COOKIE_SECURE=false
REFRESH_COOKIE_SAMESITE=lax
MAX_FAILED_LOGIN_ATTEMPTS=5
LOGIN_LOCK_MINUTES=15
PASSWORD_MIN_LENGTH=12
```

Para un despliegue con HTTPS debe utilizarse:

```text
APP_ENV=production
REFRESH_COOKIE_SECURE=true
```

En produccion la API rechaza un `JWT_SECRET` por defecto o menor de 32 caracteres.

## Impacto sobre datos existentes

- No se eliminan clientes, mascotas, planes, historiales, vectores ni modelos.
- Los usuarios existentes no quedan obligados a cambiar su contrasena.
- Las cuentas nuevas o importadas despues de la actualizacion si deben cambiar la contrasena temporal.
- Las sesiones anteriores dejan de ser validas porque el formato de token incorpora tipo y version. Los usuarios deben iniciar sesion de nuevo.

## Actualizacion desde 0.9.0

Conservar `.env`, copiar la nueva version sobre la carpeta actual y ejecutar:

```powershell
Set-ExecutionPolicy -Scope Process Bypass
.\scripts\upgrade-v0.10.ps1
```

El script:

1. Completa las variables nuevas sin sobrescribir las existentes.
2. Genera un secreto JWT local si el actual no es adecuado.
3. Reconstruye API y frontend.
4. Espera a PostgreSQL.
5. Aplica Alembic antes de arrancar la API.
6. Espera hasta seis minutos a `/health/live`.
7. Comprueba PostgreSQL mediante `/health/ready`.
8. Ejecuta las pruebas del backend.

No debe ejecutarse `docker compose down -v`.

## Reversion

Antes de actualizar, se recomienda exportar una copia de PostgreSQL. Para volver al codigo 0.9.0 hay que aplicar primero:

```powershell
docker compose -f docker-compose.yml -f docker-compose.gpu.yml run --rm --no-deps --entrypoint alembic api downgrade 0007_wakyma_integration
```

A continuacion se restauran los ficheros 0.9.0 y se reconstruyen `api` y `web`. La bajada elimina las sesiones y el registro de auditoria de esta version, pero no los datos clinicos ni los planes.

## Validacion

- 48 pruebas de backend superadas.
- Compilacion de Python completada.
- Compilacion React, TypeScript y Vite completada.
- Migracion 0008 revisada y compilada.
- No se ha podido ejecutar Docker Compose completo en el entorno de generacion; la prueba final de PostgreSQL y contenedores se realiza con `upgrade-v0.10.ps1` en Docker Desktop.

## Limitaciones conocidas

- En local se utiliza HTTP y `REFRESH_COOKIE_SECURE=false`. En cualquier publicacion externa debera habilitarse HTTPS y la cookie segura.
- El bloqueo se aplica por cuenta, no por combinacion de cuenta e IP.
- No se incluye recuperacion de contrasena por correo en este MVP.
- El registro de auditoria es append-only a nivel de aplicacion; una futura fase puede reforzarlo con permisos SQL separados o almacenamiento inmutable.
