### Revision de seguridad

- Las claves sensibles se filtran antes de escribir auditoria.
- Los cuerpos HTTP no se registran en el middleware.
- Access y refresh tokens no son intercambiables.
- Los refresh tokens se identifican mediante SHA-256.
- Los permisos de la API siguen separados en `clinic`, `staff` y `owner`.
- El propietario mantiene el filtro por pertenencia y visibilidad de eventos.
- Nginx incluye CSP y cabeceras defensivas.
- `.env` permanece excluido por `.gitignore`.

### Script de actualizacion

Se ha revisado que `upgrade-v0.10.ps1`:

- no elimina volumenes;
- conserva `.env` y solo completa variables nuevas;
- genera un JWT secreto cuando el actual es inseguro;
- espera a PostgreSQL;
- aplica Alembic antes de arrancar FastAPI;
- comprueba `/health/live`, no el endpoint inexistente `/health`;
- comprueba `/health/ready` despues del arranque;
- muestra `ps` y logs si falla;
- ejecuta la suite de backend al finalizar.

Tambien se han corregido `upgrade-v0.7.ps1`, `upgrade-v0.8.ps1` y `upgrade-v0.9.ps1` para utilizar `/health/live` durante la espera de arranque.

## Limitaciones de la validacion

El entorno de generacion no dispone de Docker Desktop, GPU NVIDIA ni un PostgreSQL persistente. Por ello no se ha ejecutado el `docker compose` completo ni la migracion 0008 sobre la base real del usuario. El script de upgrade realiza esas comprobaciones en Windows y conserva diagnosticos suficientes si se produce un fallo.
